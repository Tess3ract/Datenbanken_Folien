\begin{frame}{Iterator}
	\begin{block}{Zweck}
 		Biete eine Möglichkeit, um auf die Elemente eines zusammengesetzten Objekts sequentiell zugreifen zu können, ohne die zugrundeliegende Repräsentation offenzulegen.
 	\end{block}
\end{frame}

\note{
	Die Bezeichnungen können am Anfang etwas verwirren, aber sie sind halt sehr
	generisch. Ein Aggregat ist eine Zusammenfassung mehrere Werte. Das kann eine
	Liste sein, aber auch bspw. ein Datenbankcursor. Alle diese iterierbaren Klassen
	implementieren ein Interface (natürlich auch über mehrere Stufen möglich).  Dieses 
	Interface hat nur eine einzige Methode, die einen Iterator erzeugt. Die gemeinsame 
	Schnittstelle der Iteratoren ist das Interface Iterator. erzeugeIterator() gibt 
	einen konkreten Iterator zurück, der die Methoden zum Durchlaufen durch die Menge
	implementiert. Dazu muss der konkrete Iterator das konkrete Aggregat kennen. Der
	Klient lässt sich vom konkreten Aggregat einen konkreten Iterator erzeugen und
	benutzt diesen, um die einzelnen Elemente zu durchlaufen. Damit weiß er von der
	tatsächlichen Logik zum Durchlaufen der Elemente nichts. 	
}

\note{
	Die Implementierung des Iterator-Musters im JDK entspricht nicht ganz dem vorgestellten
	Muster. Die Iterator-Schnittstelle unterstützt die Methode remove(). Die Standardimplementierung
	wirft eine UnsupportedOperationException, wenn sie vom konkreten Iterator nicht geändert wird.
	Der Iterator kann nicht auf das erste Element gesetzt werden, was aber kein Problem ist: Man
	kann sich einfach ein neues Iteratorobjekt erzeugen lassen. Auch kann man das aktuelle Element
	nicht mit einer Methode anfordern. Aber auch das ist kein Problem, weil die Methode next() das
	aktuelle Element zurückgibt. Man muss es sich nur selbst merken. Die Methode forEachRemaining()
	erhält eine Operation, die für die restlichen Elemente ausgeführt wird. Die konkrete Iterator-Klassen
	Itr ist privat, weil der Iterator ja nur für die ArrayList arbeitet und eng mit ihr verbunden ist. 
	So kann der Iterator für die ArrayList optimiert werden. Die zweite
	Schnittstelle ListIterator stellt weitere Funktionen zur Verfügung wie z.B. das Rückwärtslaufen.
}

\begin{frame}[fragile]{Noch mehr JDK}
	\begin{exampleblock}{StandardIteratorBeispiel.java}
 		\begin{lstlisting}
Iterator<String> zauberspruchIterator = 
        zauberspruch.iterator();
while(zauberspruchIterator.hasNext()) {
    String myString = zauberspruchIterator.next();
    System.out.print(myString);
    zauberspruch.add("Und dreimal schwarzer Kater");
}
        \end{lstlisting}
 	\end{exampleblock}
\end{frame}

\note{
	Nun stellt sich die Frage, warum der Iterator Methoden anbietet zum Hinzufügen und Löschen von Elementen. Kann das
	nicht das Aggregat selbst machen? Dafür muss genau klar sein, was ein Iterator tut. Er durchläuft eine Menge von 
	Elementen und merkt sich die aktuelle Position. Er muss sich auf die zugrundeliegende Datenquelle verlassen
	können. Wenn die jetzt einfach verändert wird, gerät der Iterator in Schwierigkeiten. Deswegen wird in diesem
	Beispiel eine ConcurrentModificationException geworfen und Änderungen dürfen nur über den Iterator durchgeführt
	werden. Dann wird der Iterator nämlich informiert und er kann die Änderung berücksichtigen.
}

\note{
	Wir haben in unserem Beispiel einen externen Iterator entwickelt, d.h. der Klient steuer die Iterierung. Er ruft also
	die next()-Methode selbst auf. Beim	internen Iterator wird sie vom Iterator selbst gesteuert. 
}

\begin{frame}[fragile]{Interner Iterator}
	\begin{exampleblock}{\javapackageDrei{Iterator}}
 		\begin{lstlisting}
public interface Iterator<Music> {
	void start();
	void next();
	boolean finished();
	Music currentElement();
	void iterate(Consumer<Music> action);
} 		    
 		\end{lstlisting}
 	\end{exampleblock}
\end{frame}

\note{
	Mit Java 8 ist ein interner Iterator recht einfach zu implementieren.
}

\begin{frame}[fragile]{Interner Iterator}
	\begin{exampleblock}{\javapackageDrei{RandomIterator}}
 		\begin{lstlisting}
public class RandomIterator implements Iterator<Music> {
	private List<Music> originalList;
	private List<Music> randomList;
	private int currentPosition;
	
	public RandomIterator(List<Music> originalList) {
		this.originalList = originalList;
		start();
	}

	@Override
	public void start() {
		randomList = new ArrayList<>(originalList);
		Collections.shuffle(randomList);
		currentPosition = -1;
	}
 		\end{lstlisting}
 	\end{exampleblock}
\end{frame}

\begin{frame}[fragile]{Interner Iterator}
	\begin{exampleblock}{\javapackageDrei{RandomIterator}}
 		\begin{lstlisting}
@Override
public void next() {
    if (!finished()) {
        currentPosition++;
    }
}

@Override
public boolean finished() {
    if (currentPosition >= (randomList.size() - 1)) {
        return true;
    } else {
        return false;
    }
}
 		\end{lstlisting}
 	\end{exampleblock}
\end{frame}

\begin{frame}[fragile]{Interner Iterator}
	\begin{exampleblock}{\javapackageDrei{RandomIterator}}
 		\begin{lstlisting}
@Override
public Music currentElement() {
    if (currentPosition == -1) {
        return null;
    } else {
        return randomList.get(currentPosition);
    }
}

@Override
public void iterate(Consumer<Music> action) {
    while (!finished()) {
        next();
        action.accept(currentElement());
    }
}
 		\end{lstlisting}
 	\end{exampleblock}
\end{frame}

\note{
	Der Vorteil eines internen Iterators ist, dass der Klient weniger über die Logik wissen muss.
	Dafür sind externe Iteratoren flexibler einsetzbar, weil der Klient die volle Kontrolle
	über den Iterationsprozess hat. Welche Variante die bessere ist, hängt vom Anwendungsfall ab. 
}

\frameWithImageOnly{\BILDERPFAD/iterator8.jpg}
\note{
	Was machen wir jetzt mit Änderungen, die während des Iterierens durchgeührt werden sollen? Bei 
	selbst geschrieben Iteratoren muss man sich darüber natürlich Gedanken machen, wenn man so
	etwas erlauben möchte. Wenn man es verbieten möchte, geht dies natürlich nur mit einer 
	ordentlichen Kapselung. Auf keinen Fall sollte man das Problem ignorieren, denn es könnte zu
	Laufzeitfehlern führen, die im Extremfall logische Fehler zur Folge haben. Im einfachsten Fall 
	werden die Elemente einfach im Iterator kopiert und Änderungen werden nur an dieser internen
	Liste durchgeführt. Das scheidet aus Performancegründen bei vielen Datensätzen aus. Man könnte -- wie 
	bei der ArrayList -- die entsprechenden Methoden im Iterator unterbringen. Dann müssen Iterator und
	Aggregat allerdings miteinander sprechen, denn es könnte ja sein, dass verschiedene Klienten
	zur gleichen Zeit über die Daten laufen und Änderungen vornehmen.  
}

\frameWithImageOnly{\BILDERPFAD/iterator9.jpg}
\note{
	Mit Iteratoren kann man auch durch komplexere Datenstrukturen laufen wie bspw. Bäume, die mithilfe
	eines Kompositummusters erstellt wurden. Dann muss sich der Iterator nicht nur die Position des
	aktuellen Elements merken, sondern auch den Pfad dorthin. Man könnte einen solchen Iterator auch
	rekursiv lösen. Dann müsste er sich nicht den gesamten Pfad merken, sondern der Pfad ergibt sich
	aus dem Aufrufstack.
